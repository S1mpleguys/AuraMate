# 任务拆分与里程碑

## 结论

按比赛节奏，建议分成五条并行主线：板级 bring-up、传感器链路、音频与交互、云端与算法、展示与答辩。所有任务都围绕同一骨架推进，避免临近答辩时重构目录。

## 角色建议

- 方向 A：板级与驱动
  负责 `components/board/`、`components/drivers/`

- 方向 B：服务与应用流程
  负责 `components/services/`、状态机、训练流程

- 方向 C：云端与算法
  负责动作评分、Prompt/RAG、HTTP/MQTT 接口

- 方向 D：展示与内容
  负责 UI 文案、演示流程、PPT、视频和答辩脚本

## 周计划

### 第 1-2 周

- 跑通 `idf.py build/flash/monitor`
- 确认板级引脚和外设型号
- 读通 I2C 传感器
- 跑通 I2S 录音和播放
- 点亮 LCD 和 RGB

### 第 3-4 周

- 接通 Wi-Fi
- 打通 HTTP 或 MQTT
- 建立音频上云链路
- 建立 IMU 数据采集脚本
- 形成第一版动作模板数据

### 第 5-6 周

- 完成姿态采样与预处理
- 完成 DTW/相似度评分最小版本
- 完成环境阈值提醒逻辑
- 完成第一版语音问答闭环

### 第 7-8 周

- 联调整机演示流程
- 打磨 UI 和语音文案
- 输出训练报告或演示看板
- 完成外壳、PPT、视频和答辩材料

## Git 协作建议

- 驱动改动和算法改动不要堆在同一分支。
- 每次提交只解决一类问题。
- 合并前必须重新 `idf.py build`。
- 新增假设、引脚、协议定义时先更新文档再提代码。
